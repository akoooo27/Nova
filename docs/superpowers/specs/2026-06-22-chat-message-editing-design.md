# Chat Message Editing Design

## Goal

Allow a user to edit one of their user messages on the active conversation path. Editing does not mutate the original message or remove its descendants. It creates a new user-message sibling, starts a new assistant response beneath that sibling, and makes the new response the active branch.

Nova will expose this as a focused edit operation rather than reproduce ChatGPT's generic `action: "next"` request envelope. The existing `ChatThread` tree already owns parent lookup, sibling placement, branch selection, and message identity generation.

## Current Architecture Fit

Nova already represents a conversation as a branching message tree:

- `ChatThread` is the aggregate root.
- `ChatMessage.ParentMessageId` forms the tree.
- `ChatThread.CurrentMessageId` identifies the active branch head.
- `SiblingIndex` records creation order within a sibling group.
- `ChatThread.EditUserMessage` creates an immutable user-message sibling under the target's parent and moves the head.
- `ChatThread.BeginAssistantMessage` creates the generating assistant child and moves the head again.
- `ChatTurnOrchestrator` and `ContextBuilder` generate from the selected root-to-head path.
- `TurnRequested` is published through the MassTransit EF outbox before `SaveChangesAsync`.

The feature therefore requires application and HTTP orchestration plus two domain guards. It does not require message mutation, node hiding, deletion, or a persistence migration.

## Selected Approach

Add a dedicated FastEndpoints endpoint backed by a `Mediator` command and handler:

```http
POST /v1/chats/{chatId}/messages/{messageId}/edit
```

The route `messageId` identifies the existing user message being edited. The aggregate derives the sibling parent from that message's existing `ParentMessageId`; the client does not submit tree placement.

The handler follows the established send/regenerate orchestration:

1. Validate and create the typed user, chat, message, content, and model identifiers.
2. Ensure the requested model is usable for the selected generation options.
3. Load the owner-scoped `ChatThread`.
4. Ask the aggregate to create the edited user sibling.
5. Ask the aggregate to create a generating assistant child beneath it.
6. Publish one `TurnRequested` for the new assistant message.
7. Save through the existing unit of work and outbox transaction.
8. Return the new user and assistant message identifiers.

This keeps transport concerns in the endpoint, orchestration in the application layer, and tree invariants in the domain.

## API Contract

### Request

```http
POST /v1/chats/{chatId}/messages/{messageId}/edit
Content-Type: application/json
```

```json
{
  "message": "Updated text",
  "modelId": "0b5d0000-0000-0000-0000-000000000000",
  "forceUseSearch": false
}
```

Fields:

- `message` is required text and follows the existing chat message length limits.
- `modelId` is required. User messages do not store model IDs, and one user message may have assistant children produced by different models.
- `forceUseSearch` is optional and defaults to `false`.

The request intentionally excludes:

- `parentMessageId`, because the aggregate derives it from the target message.
- A client-generated ID for the edited message or assistant response.
- ChatGPT-specific `action`, `messages`, timezone, client metadata, buffering, and encoding fields.
- Multipart content and attachments.

### Success

Success returns `202 Accepted` with the existing `TurnStartedResponse`:

```json
{
  "chatId": "...",
  "userMessageId": "...",
  "assistantMessageId": "..."
}
```

`userMessageId` is the new edited sibling. `assistantMessageId` is its new generating child. Both IDs are generated by the server and belong to the existing chat.

The chat title remains unchanged, including when the root user message is edited.

## Domain Behavior

### Active-path rule

Only a user message on the active path may be edited. The active path is the ancestor chain from `CurrentMessageId` to the root, including both endpoints.

`ChatThread.EditUserMessage` will validate that:

1. The target exists in this aggregate.
2. The target role is `User`.
3. The target belongs to the active path.
4. The active path contains no assistant message in `Generating` status.

Keeping these checks in the aggregate prevents application callers from creating edits that violate conversation state.

### Branch creation

For a valid target, `EditUserMessage`:

1. Creates a completed user message with the submitted content.
2. Reuses the target's `ParentMessageId`.
3. Assigns the next `SiblingIndex` in that parent group.
4. Adds the sibling without modifying the target or any descendants.
5. Moves `CurrentMessageId` to the sibling.

Editing a root user message is allowed. The new sibling also has `ParentMessageId = null` and receives the next root sibling index.

The handler immediately calls `BeginAssistantMessage` with the new user sibling as parent. That operation creates a generating assistant node and moves `CurrentMessageId` to it. The original path becomes inactive naturally because rendering follows the new head's ancestry. No explicit hidden/deleted marker is needed.

### Provenance

No `EditedFromMessageId` field is added. The immutable original, shared parent, and sibling order already encode the edit as an alternate branch. This avoids redundant state and a schema migration.

### Temporary chats

Temporary chats support editing through the identical aggregate and application flow. Their existing cleanup lifecycle is unaffected.

## Application Components

Add an `EditMessage` command slice alongside `SendMessage` and `RegenerateMessage`:

- `EditMessageCommand`
- `EditMessageCommandValidator`
- `EditMessageHandler`

The command contains:

```text
ChatId
MessageId
Message
LlmModelId
GenerationOptions
```

The handler uses the same dependencies and ordering as `SendMessageHandler`:

- `IUserContext`
- `IChatRepository`
- `ILlmProviderRepository`
- `IMessageBus`
- `IUnitOfWork`
- `IDateTimeProvider`

The handler reads `UtcNow` once and uses that value for both newly created nodes. It publishes `TurnRequested` before `SaveChangesAsync` so the existing MassTransit outbox keeps the aggregate update and turn request atomic.

## Data Flow

```text
POST /v1/chats/{chatId}/messages/{messageId}/edit
  -> FastEndpoints endpoint maps route and body
  -> EditMessageCommand
  -> EditMessageHandler
       validates typed values
       validates model usability
       loads owner-scoped ChatThread
       thread.EditUserMessage(messageId, content, now)
       thread.BeginAssistantMessage(editedUserMessageId, modelId, now)
       publishes TurnRequested(newAssistantMessageId, options)
       SaveChangesAsync
  -> 202 Accepted + TurnStartedResponse

... asynchronously ...
TurnRequested
  -> ChatTurnOrchestrator
  -> ContextBuilder walks from the new assistant's parent to the new active root
  -> IAgentRunner streams tokens
  -> CompleteAssistantMessage or FailAssistantMessage
```

Because the edited node is a sibling under the original target's parent, `ContextBuilder` automatically excludes the original target and all of its descendants from the new generation context.

## Error Handling

Errors continue through `ErrorOr` and `CustomResults.Problem`:

- Invalid route IDs, empty or oversized text, or an empty model ID: `400 Bad Request`.
- Requested model is missing, disabled, or cannot support forced search: existing model-usability `400 Bad Request`.
- Owner-scoped chat not found: `404 Not Found`.
- Target message not found in the chat: `404 Not Found`.
- Target is not a user message: existing `Chat.EditTargetMustBeUser`, `409 Conflict`.
- Target exists but is not on the active path: new `Chat.EditTargetNotOnActivePath`, `409 Conflict`.
- The active path contains a generating assistant: new `Chat.CannotEditWhileGenerating`, `409 Conflict`.

The domain checks role before active-path eligibility so an assistant target consistently reports that only user messages are editable.

No edit-specific locking, row version, or idempotency key is introduced. The command uses the project's existing persistence and concurrency behavior.

## Testing

Test work is explicitly in scope.

### Domain tests

- Edit an active-path non-root user message.
- Edit the active root user message.
- Preserve the original target and descendants.
- Assign the correct parent and next sibling index.
- Move the head to the edited sibling.
- Reject an assistant target.
- Reject a user target on an inactive branch.
- Reject editing while the active path contains a generating assistant.
- Allow editing after an assistant response completed.
- Allow editing after an assistant response failed.

### Application tests

- Create the edited user sibling and generating assistant child.
- Use the required requested model.
- Forward `forceUseSearch` through `TurnGenerationOptions`.
- Publish exactly one `TurnRequested` for the new assistant ID.
- Return the new user and assistant IDs.
- Reject an unusable model without publishing or saving.
- Propagate chat-not-found and domain edit errors without publishing.

### Validator tests

- Reject empty chat, message, and model IDs.
- Reject empty and oversized message content.

No endpoint integration-test framework is introduced solely for this feature.

## Alternatives Considered

### Selected: dedicated edit endpoint and command

This matches Nova's FastEndpoints and `Mediator` conventions, mirrors the dedicated regenerate operation, and leaves tree invariants inside `ChatThread`.

### Extend `SendMessage` with an edit target

This would reduce the number of command files but mix normal append semantics with sibling-branch semantics. Conditional validation and orchestration would make the command harder to understand and evolve.

### Reproduce ChatGPT's generic action envelope

Accepting `action`, a messages array, `parent_message_id`, and client-generated IDs would make the client responsible for domain placement and allow contradictory target/parent combinations. Nova's dedicated route and aggregate already provide a smaller, safer contract.

## Out of Scope

- Mutating or deleting the original message.
- Explicitly hiding old nodes.
- Editing assistant messages.
- Editing user messages on inactive branches.
- Attachments or multipart message content.
- Changing or regenerating the chat title.
- Adding edit provenance columns.
- Introducing a new concurrency or idempotency system.
- Changing MassTransit versions or replacing the existing `Mediator` package family.
